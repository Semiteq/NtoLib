# Замена FB-инстанса между runtime-циклами

## Описание проблемы

MasterSCADA **пересоздаёт инстанс функционального блока** при переходе между Runtime и Design. Объект, в который писали поля во время предыдущего цикла, выбрасывается; вместо него создаётся новый инстанс, десериализованный из снапшота, снятого **до** запуска отложенных делегатов.

**Симптом для FB с отложенным выполнением:** значения, записанные в поля FB после `ToDesign()` (например, статус операции, сообщение об ошибке), исчезают. В следующем runtime-цикле новый инстанс начинает с дефолтных или снимочных значений; output-пины не отражают результат отложенного выполнения.

## Причина

Замена инстанса происходит на native COM-уровне (или через `IPersistStream`-сериализацию) и **не видна** в декомпилированных .NET-исходниках MasterSCADA.

Снапшот, из которого восстанавливается инстанс, фиксируется на ранней стадии shutdown-последовательности — до того, как отложенный делегат, запостенный из `ToDesign()`, успеет выполниться (см. [`06-runtime-tree-modification-forbidden.md`](06-runtime-tree-modification-forbidden.md) про последовательность завершения).

## Что выживает, а что нет

| Объект | Сохраняется после замены инстанса? |
|--------|-----------------------------------|
| Поля FB (включая `[Serializable]`) | Нет — берутся из старого снапшота |
| Output-пины | Нет — состояние пинов привязано к инстансу |
| Closure-захваченные ссылки на heap-объекты (`IProjectHlp`, service, plan, logger) | Да — это независимые объекты, делегат держит их через замыкание |

Делегат, запостенный из `ToDesign()`, исполняется после уничтожения старого инстанса. Все его записи в поля старого FB уходят в никуда. Доступ к нужным объектам сохраняется только через closure.

## Workaround: файловое логирование

Для FB с отложенным выполнением результат должен писаться **на диск**, а не в поля FB или output-пины. Стандартное место — `C:\DISTR\Logs\<fb-name>.log` через Serilog.

Минимальная схема (без DI):

1. В `InitializeRuntime()` создать `Serilog.Core.Logger` (sink: File + Debug).
2. Передать `ILogger` в сервис через конструктор.
3. В отложенном делегате логгер захватывается замыканием — он живёт независимо от инстанса FB.
4. Disposed логгера должен делать сам делегат после завершения работы (так как `CleanupRuntime()` старого инстанса уже не вызовется правильно).

**Конфигурация Serilog:**

- `path: C:\DISTR\Logs\<fb-name>.log`
- `rollingInterval: RollingInterval.Infinite`
- `fileSizeLimitBytes: 5 * 1024 * 1024`
- `rollOnFileSizeLimit: true`
- `retainedFileCountLimit: 5`
- `shared: true`
- `outputTemplate: "{Timestamp:O} [{Level:u3}] {Message:lj}{NewLine}{Exception}"`

## Чего нельзя делать

- Хранить результат отложенного выполнения в поле FB (с `[Serializable]` или без) — поле принадлежит старому инстансу и будет потеряно.
- Писать результат в output-пин из отложенного делегата — пин привязан к старому инстансу.
- Полагаться на `[NonSerialized]` для сохранения состояния между циклами — поле всё равно ушло вместе со старым инстансом.
