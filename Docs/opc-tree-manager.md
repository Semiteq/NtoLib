# OpcTreeManager — менеджер дерева OPC

> **TL;DR:** компонент выполняет перестроение поддерева OPC UA (FB) в проекте MasterSCADA в соответствии с выбранной конфигурацией. Файл `config.yaml` задаёт требуемую структуру (любой глубины вложенности), а `tree.json` содержит эталонное состояние узлов и их связей.
> По сигналу `Execute` формируется план изменений: удаляются лишние узлы (с разрывом связей), добавляются отсутствующие (с восстановлением связей), а также ограничивается состав дочерних узлов.
> По сигналу `ExecuteSnapshot` текущее состояние сохраняется в `tree.json`.
> Сканирование выполняется в Runtime, исполнение — отложенно в Design mode. Результаты фиксируются в логе.

---

## 1. Назначение

Компонент предназначен для мультипроектной эксплуатации, когда один крупный проект (порядка 1000 тегов) разделяется на несколько меньших (около 500 тегов) для отдельных установок.

Ручное управление составом OPC-группы (включение/отключение узлов и восстановление их связей) является трудоёмким и подверженным ошибкам процессом. Данный функциональный блок автоматизирует эти операции, выполняя их детерминированно на основе конфигурационных файлов.

---

## 2. Интерфейс

### Входы

| Имя             | Тип  | Описание                                                                |
| --------------- | ---- | ----------------------------------------------------------------------- |
| Execute         | bool | Переход `0 → 1` запускает сканирование и формирование плана перестройки |
| Cancel          | bool | Переход `0 → 1` отменяет сформированный план                            |
| ExecuteSnapshot | bool | Переход `0 → 1` сохраняет текущее состояние группы в `tree.json`        |

### Выходы

| Имя       | Тип  | Описание                          |
| --------- | ---- | --------------------------------- |
| IsPending | bool | Признак наличия отложенного плана |
| Failed    | bool | Признак ошибки последней операции |

### Параметры

| Свойство             | Значение по умолчанию                               | Описание                     |
| -------------------- | --------------------------------------------------- | ---------------------------- |
| Целевой проект       | `MBE`                                               | Имя проекта из `config.yaml` |
| Путь к OPC FB        | `Система.АРМ.OPC UA Siemens`                        | Полный путь к OPC FB         |
| Имя группы OPC       | `MBE`                                               | Имя управляемой группы       |
| Путь к `tree.json`   | `C:\DISTR\Config\OpcTreeManager\tree.json`          | Эталонный снапшот            |
| Путь к `config.yaml` | `C:\DISTR\Config\OpcTreeManager\config.yaml`        | Конфигурация проектов        |
| Путь к логу          | `C:\DISTR\Logs\OpcTreeManager\opc-tree-manager.log` | Файл лога                    |

## 3. Внешние файлы

### 3.1. `tree.json` — эталонное состояние

Файл содержит снимок структуры группы и используется как источник для восстановления узлов и связей.

* Формируется через `ExecuteSnapshot`
* При выполнении `Execute` используется только для чтения

Структура:

* ключ — имя узла верхнего уровня
* `scadaItem` — описание узла
* `links` — список внешних связей

Особенности:

* повторяющиеся связи удаляются
* тип связи (`iconnect`, `directPin`, `directPout`) определяет способ подключения

---

### 3.2. `config.yaml` — описание целевых конфигураций

Файл задаёт состав узлов для каждого проекта.

Описывать состав можно двумя способами:

1. **Указать узел без вложенных элементов**
   Узел сохраняется полностью вместе со всем поддеревом.

Например:

```yml
projects:
  MBE:
    - Cameras # Будут использоваться все вложенные элементы в Cameras
```

2. **Узел с дочерними элементами**
   Узел сохраняется, но его дочерние элементы ограничиваются указанным списком.

```yml
projects:
  MBE:
    - Cameras:
      - Camera1 # Будут использоваться только указанные вхождения
      - Camera3 # Отсутствующие в перечислении - будут отброшены
```

Правила:

* вложенность не ограничена

```yml
projects:
  MBE:
    - Cameras:
      - Camera1:
        - Subchamber1
```

* имена должны совпадать с именами в проекте и `tree.json`
* допускается смешивание типов

```yml
projects:
  MBE:
    - Cameras:
      - Camera1:
        - Subchamber1
      - Camera2
```

---

### 3.3. Полнота снапшота

Файл `tree.json` должен содержать объединение всех узлов, используемых во всех проектах.

Если узел отсутствует:

* он не будет создан при выполнении
* в логе появится запись о пропуске

При добавлении новых узлов:

1. обновить OPC сервер 
2. выполнить `ExecuteSnapshot`

## 4. Выполнение `Execute`

### Этап Runtime

1. Загрузка `config.yaml`
2. Загрузка `tree.json`
3. Построение плана редактирования
4. Установка `IsPending = true`. В случае ошибки или отсутствия изменений `IsPending = false`

---

### Этап Design mode (отложенное выполнение)

1. Ожидание выхода из Runtime (до 20 секунд)
2. Применение плана:
   * удаление лишних узлов и связей
   * добавление отсутствующих узлов
   * обновление состава дочерних элементов
3. Применение изменений к OPC серверу
4. Восстановление связей для новых узлов

Особенности:

* изменения применяются рекурсивно
* существующие узлы не переподключаются
* замена состава происходит атомарно на каждом уровне дерева отдельно; отката всего плана при сбое в середине нет — см. 6.2

---

## 5. Выполнение `ExecuteSnapshot`

Операция выполняется немедленно:

1. Сканируется текущая группа
2. Собираются связи всех узлов
3. Результат записывается в `tree.json`

Используется для формирования эталонного состояния.

---

## 6. Обработка ошибок

### Ошибки валидации (останавливают выполнение)

* отсутствуют или повреждены `config.yaml` / `tree.json`
* не найден целевой проект
* не найден OPC FB или группа
* некорректные параметры

Результат:

* операция отменяется
* `Failed = true`
* запись в лог

---

### Ошибки выполнения (не останавливают процесс)

* не найден пин
* ошибка `Disconnect` / `Connect`
* частичные сбои

Результат:

* ошибка фиксируется в логе
* выполнение продолжается

Откат изменений не предусмотрен.

---

## 7. Режим работы

Изменение структуры проекта в Runtime запрещено, поэтому используется отложенное выполнение:

1. В Runtime формируется план
2. После перехода в Design mode выполняется применение
3. Таймер ожидает до 20 секунд
4. При превышении времени выполнение отменяется

Важно:

* после остановки Runtime не изменяйте проект вручную в течение ~20 секунд
* результаты доступны только в логе

---

## 8. Логгирование

* **Файл:** задаётся параметром `LogFilePath`
* **Ротация:** Ротация: по размеру, 5 МБ на файл, удержание 5 файлов
* **Библиотека:** Serilog
* **Содержимое:**

  * загрузка конфигурации
  * список изменений
  * операции со связями (Debug)
  * итоговая статистика (Information)
